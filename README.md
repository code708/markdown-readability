# @code708/markdown-readability

Custom markdownlint rules to improve readability of long markdown documents.

## Installation

For programmatic use with `markdownlint`:

```bash
npm install --save-dev markdownlint @code708/markdown-readability
```

For CLI use with `markdownlint-cli`:

```bash
npm install --save-dev markdownlint-cli @code708/markdown-readability
```

## Usage

### Documentation and Demonstration

Consumer documentation and examples:

- [Rules overview](doc/rules-overview.md)
- [Example violating every rule](example/violates-all-rules.md)
- [Example respecting all rules](example/respects-all-rules.md)

Try the demo:

```bash
# Lint example/violates-all-rules.md
node example/demo.js

# Lint example/respects-all-rules.md
node example/demo.js --no-violation
```

### Programmatic Usage

```javascript
const markdownlint = require('markdownlint');
const customRules = require('@code708/markdown-readability');

const options = {
  files: ['**/*.md'],
  customRules,
  config: {
    default: true,
    'C708-001': true,
    'C708-002': true,
    'C708-003': true,
    // ...
  },
};

const result = markdownlint.sync(options);
console.log(result.toString());
```

### `markdownlint-cli`

Create a `.markdownlint.json` file in your project root:

```jsonc
{
  "default": true,
  "C708-001": true,
  "C708-002": true,
  "C708-003": true
  //...
}
```

Then lint your Markdown with the package name as the custom rules source:

```bash
npx markdownlint --config .markdownlint.json --rules @code708/markdown-readability '**/*.md'
```

## Lean Runtime Package Policy

The published npm package only includes runtime files. Repository documentation, examples, and demo assets stay in the Git repository and are not shipped in the npm tarball.

## Merge Policy

Use rebase merges only. Squash merges are not allowed because the release workflow depends on the pull request commits being preserved on `main` as a linear history.

This approach does not scale well with parallel pull requests and is not intended to at this point in time. Pull requests should be created one at a time so version bumps and release boundaries stay unambiguous.

## Vulnerabilities

`npm audit` currently reports 7 advisories (4 moderate, 3 high) — **all dev-only**. The published runtime tarball ships only runtime code (`index.js`, `src/`) plus the standard metadata files (`package.json`, `README.md`, `LICENSE`), none of which import any flagged package. Consumers of this library are not exposed.

### Current advisories

| Package            | Severity     | Source path                                              | Why it's not exploitable here                                                         |
| ------------------ | ------------ | -------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `glob`             | high         | `markdownlint-cli` → `glob`                              | Command injection only via `glob -c/--cmd` CLI usage; we invoke glob as a library.    |
| `minimatch`        | high         | `markdownlint-cli` → `glob` → `minimatch`                | ReDoS only with attacker-controlled patterns; patterns here come from project config. |
| `markdown-it`      | moderate     | `markdownlint` → `markdown-it`                           | ReDoS only with attacker-controlled markdown; we lint our own docs.                   |
| `brace-expansion`  | moderate     | `markdownlint-cli` → `minimatch@9` → `brace-expansion@2` | Process hang only with attacker-controlled patterns.                                  |
| `js-yaml`          | moderate     | `markdownlint-cli` → `js-yaml`                           | Prototype pollution via YAML merge keys; we don't parse untrusted YAML.               |
| `markdownlint-cli` | (transitive) | direct devDep                                            | Flagged because of `glob` and `markdownlint` ranges above.                            |
| `markdownlint`     | (transitive) | direct devDep + peerDep                                  | Flagged because of `markdown-it` range above.                                         |

Release pull requests are generated by `cicd/pull-request.js`, a small repo-local Node script with no runtime dependencies.

### What to watch

- **Any of these advisories upgrades to "critical"** — re-evaluate immediately.
- **A new advisory appears against a runtime path** (anything imported by `index.js` or `src/`) — must fix before the next release; consumers would inherit it.
- **A new advisory affects `markdownlint`'s `peerDependencies` range** — bump the peer range; consumers using the same range will be flagged.

### When to replace

- **`markdownlint-cli`**: replace or upgrade past `0.44.0` if the `glob`/`minimatch` advisories ever become exploitable in library mode (currently they aren't). `npm audit fix --force` jumps to `0.48.0` (breaking).
- **`markdownlint`**: bump past `0.40.0` if `markdown-it` advisories escalate. This is a peerDep — coordinate with consumers, since bumping it is a SemVer breaking change for downstream.
- **release tooling**: keep `cicd/pull-request.js` dependency-free unless the release flow grows enough to justify a maintained package.
